### 2-1. 시스템 프롬프트 (변환 규칙)

**핵심 지시 위치:** js/app.js:11815–12053 (`buildConversionPrompt()` 반환 문자열)

**최우선 규칙 6개** (11854–11866):
1. 원문 텍스트 100% 보존 — 글자 하나 변경/추가/삭제 금지
2. 번호·기호 체계 100% 보존 (1., 가., ○, ※ 등)
3. 기존 구조 요소 보존 (table, ul/ol, heading 레벨)
4. 환각(hallucination) 절대 금지
5. 내용 요약·축약 금지 + 표 생략 금지
6. 중첩 테이블 위치 절대 보존 — 셀 안 테이블을 바깥으로 빼내는 것 금지

**섹션 감지 규칙** (11879–11892):
- 대분류만 `<h2>` (회사소개, 모집부문, 담당업무 등 11개)

**HTML 태그 규칙** (11894–11941):
- rowspan/colspan 정밀 계산 (행/열 포함 개수, 값 검증)

- 링크 변환: `<a href="URL" target="_blank" class="noko">URL</a>`
- 텍스트 배경색 필수 보존: `<span style="background-color: #색상;">...</span>`

**특수 처리** (11942–12070):

**PDF 입력 특수 처리** (11872–11877):
- 끊어진 줄 → 문단 병합
- 표 텍스트 풀림 → `<table>` 재구성

**현재 문제:** 프롬프트가 AI에게 "원문 텍스트 + 구조"를 **모두 생성**하도록 요구. 텍스트 추가/삭제/변경으로 환각이 발생할 여지가 있음.

## 조사자 관찰 — 재설계 시 주의 지점

### 1. 원문 기반 강제 복원 vs 새 아키텍처

**현황:**
- `fixStarIndent()` (line 4997–5005): AI가 별표를 제거한 경우, 원문 snippet으로 찾아서 별표를 강제로 삽입
- `fixMiddleDots()` (line 4892–4950): 깨진 "틀" 문자를 원문에서 "·"가 있던 위치로 찾아 복원
- `fixLegalArticleStructure()` (line 5173–5196): 원문의 "N의M" 패턴으로 번호를 강제 교정

**재설계 영향:**
새 아키텍처에서 텍스트가 원문에서 그대로 복사되면, AI가 이들 요소를 삭제/변경할 여지 자체가 없어 이 함수들은 **불필요해질 수 있음**. 마이그레이션 대상으로 표시하고 새 아키텍처 검증 후 제거 검토.

### 2. 중첩 테이블 — 최우선 규칙인데 구조만으로 완전히 표현 불가

**현황:**
- 최우선 규칙 6번: "중첩 테이블을 절대 바깥으로 빼내지 마세요"
- `renestOrphanedTables()`는 텍스트 기반 유사도로 셀과 테이블을 재매칭 (불완전)

**제안:** 새 아키텍처에서 AI에게 "구조"만 요청할 때, 원문 테이블 구조(rowspan/colspan)를 정답 참고로 제공. 후처리 `renestOrphanedTables()`는 검증용으로만 남김.

### 3. 색상/하이라이트 보존 — AI의 임의 추가가 빈번

**현황:**
- 프롬프트에 "원문에 색상이 없으면 색상 추가 금지"
- `fixArbitraryColors()`가 AI 추가 색상 제거
- 그럼에도 검증에서 색상 보존 위반 자주 감지

**제안:** 전처리 단계에서 원문의 모든 색상(style 속성 background-color, color) 추출 + 좌표 매핑. AI는 구조만, 코드가 색상을 병합.

### 4. 법률 조항 N의M + 별표 들여쓰기 — 원문 기반 계산 필수

**현황:**
- N의M 번호: 원문의 "6의1.", "6의2." 텍스트를 정확히 찾아 텍스트로 보존 (CSS 번호 자동 생성 불가)
- 별표 들여쓰기: 원문의 `*`, `**`, `***` 개수 세어 nbsp 개수 계산 (구조 결정 불가)

**제안:** 전처리에서 원문의 "N의M" 패턴과 별표 수를 메타데이터로 추출. AI는 `<ol>/<li>` 구조 제공, 코드가 텍스트 + 들여쓰기 병합.

### 5. 테이블 rowspan/colspan 정밀 계산 — AI가 자주 실패

**현황:**
- 프롬프트에 rowspan/colspan 정밀 계산 규칙 상세 설명
- 검증에서 rowspan이 표 높이를 초과하지 않는지 확인
- 그럼에도 AI는 rowspan 값을 자주 과대/과소 설정 (2026-04-14 발단 사건)

**제안:**
- **Option A (권장):** 원문 테이블에서 rowspan/colspan을 코드가 계산. AI는 "카테고리 셀 경계" 등 로직만 결정.
- **Option B:** AI가 rowspan/colspan 생성, 원문과 비교 검증 + 모듈로 보정.
- Option A (구조/텍스트 완전 분리)가 더 안정적.

## 최종 평가

현재 아키텍처는 AI에게 "텍스트 생성"과 "구조 결정"을 **동시에 요구**하므로, hallucination 위험이 높음.

제안된 분리 아키텍처("구조만 AI, 텍스트는 코드")는 이 리스크를 대폭 감소시킬 것으로 예상되나, **원문 메타 추출 함수군의 개발 복잡도 증가** 및 **후처리 검증 로직 재설계**가 필수.
